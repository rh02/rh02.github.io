Zabbix学习笔记|Zabbix基本概念

标准监控系统所具有的基本功能：

  • 监控主机中的各项指标：

    • 主机：
      • 系统指标：CPU，Memory，IO（Disk，Network）
        • CPU：sys,usr,idle,…
        • Memory: total,used,free,cached,buffer,shm,…
        • IO :
      • 业务指标：
        • 针对某一服务的状态进行监控，与服务相关的监控项，例如监控nginx的健康状态。

  • 监控系统的模型为：

    • server/agent

  • 监控系统监控的三种方式：

    • 服务器<——> (agent代理)操作系统主机
    • 服务器<——> （IPMI–inter智能接口）只有硬件没有操作系统的主机
    • 服务器<——-> (SNMP:简单网络管理协议)交换机、路由器 （IPMI是集成在硬件级的）
    • 还有其他的一些方式

  • 第一阶段：数据采集

    • 对于关键指标数据采集，采集数据的时间间隔要小一些
    • 对于非关键指标但又不得不采集的，采集数据的时间间隔可以长一些

  • 第二阶段：存储数据

    • 采集的数据没必要全部保存，将采集来的数据按时间间隔进行整合，做成趋势数据，展现出来。
    • 聚合数据：将一段时间内的数据的最大值和最小值，根据间隔时长，取平均数据，这就是聚合数据。
    • 采集的数据的存储方式：
      • rrd(cacti）：round-robin database（轮转存储数据）
        • 固定一个大小的存储数据空间，当存满后，新数据将最旧的数据覆盖掉，一直循环，这样就可以只保存固定时间长度的采集数据
      • 时序存储数据库：
        • influxdb：

  • 第三阶段：报警

    • 事先编辑好短信服务，或者邮件服务
    • 当监控系统监控的数据达到一定的级别的时候，监控系统触发第三方的短信服务或邮件服务，按照固定的模板发送报警信息。

  • 第四阶段：展示数据

    • 将采集的数据进行整合，以靓丽的图形的方式展现出来。

    • Nagios：很好的报警系统，但是没有数据存储功能

    • Cacti: cron+ SNMP + mysql 周期性计划任务，使用SNMP这个协议采集数据，将数据放入mysql中。展示功能不错，但是报警功能不好。

    • zabbix:

      • 数据采集：agent，SNMP，IPMI，JMX（支持的数据采集方式）
      • 存储数据：MySQL，PGSQL
      • 报警：email，script(微信、短信)‘报警前可以执行远程命令进行故障自动恢复，恢复不了再进行报警’
      • 展示：简单图（一个界面展示一张图），图形，screen（一屏上展示多张图），slide show，map（将数据绘制成连接线）。

    • 可以采用的几种监控方式：

      • statsd+ influxdb + grafana
        • #statsd：数据采集器，采集数据
        • #influxdb：数据存储，将statsd采集的数据放入
        • #grafana：提取数据进行展示
      • promethues + grafana
        • #适合大规模监控系统
        • #promethues：采集数据并存储数据
        • #grafana：展示数据
      • graphite + grafana

  • zabbix的组成及作用：

    1. zabbix server: 采集收集监控数据，存储到数据库中
    2. database: 提供数据存储业务
    3. zabbix agent：放置在被监控端，向zabbix Server提供采集到的数据，有主动模式和被动模式.
      • 被动模式:zabbix server主动到agent上获取数据;
      • 主动模式：zabbix Server接收来自agent发送来的采集数据。
    4. zabbix web-gui: zabbix管控接口，通过php程序到数据库中将数据取出来，进行展示。
    5. zabbix sender：放置在被监控主机，用于向zabbix Server发送采集数据，通过命令行命令执行，主要用于测试
    6. zabbix get：放置在zabbix Server，通过命令行执行，去获取监控端的采集数据，主要用于测试。

  • Zabbix组件概述：（可能与上面的介绍有重复介绍）

    1. zabbix Server：负责接收agent发送的报告信息的核心组件，所有配置、统计数据及操作数据均由其组织进行。
    2. Database Storage：专用于存储所有配置，以及由zabbix收集的数据。
    3. Web interface：zabbix的GUI接口，通常与Server运行在同一台主机上，也就是zabbix Web-GUI。
    4. Proxy：可选组件，常用于分布监控环境中，代理Server收集部分被监控端的监控数据，并同一发往Server端。
    5. Agent：部署在被监控主机上，负责收集本地数据并发往Server端或proxy端。

  • zabbix常用术语：

    • 主机（host）：被监控的网络设备，可由IP或DNS名称指定
    • 主机组(host group)：主机的逻辑容器，可以包含主机和模板，但同一个组内的主机和模板不能相互链接；主机组通常在给用户或者用户组指派监控权限时使用
    • 监控项（item）：一个特定监控指标的相关的数据，这些数据来自于被监控对象，item是zabbix进行数据收集的核心，没有item，将没有数据，相对某监控对象来说，每个item都有‘key’进行标识。
    • 触发器（trigger)：一个表达式，用于评估某监控对象的某特定item内所接收的数据是否在合理范围内，即阈（yu）值；接收到的数据量大于阈值，触发器状态将从OK转变为problem，当数据量再次回归到合理范围内时，其状态将从problem转变为OK。
    • 事件（event）：发生的一个值得关注的事情，例如触发器的状态转变，新的agent或重新上线的agent的自动注册等
    • 动作（action）：指定对于特定事件事先定义的处理方法，通常包含操作和条件。
    • 报警升级（escalation）：发送报警或执行远程命令的自定义方案，如每隔5分钟发送一次警报，共发送5次等。
    • 媒介（media）：发送通知的手段或通道，如Email，Jabber或SMS等
    • 通知（notification）：通过选定的媒介向用户发送的有关某事件的信息。
    • 远程命令（remote command）：预定义的命令，可在被监控主机处于某特定条件下时自动执行。
    • 模板(template)：用于快速定义被监控主机的预设条目集合，通常包含了item、trigger、graph、screen、application以及low-level discovery rule;模板可以直接至单个主机
    • 应用（application）：一组item的集合
    • web场景(web scennario): 用于检测web站点可用性的一个或多个http请求
    • 前端（frontend）：Zabbix的web接口

  • zabbix监控系统的整体流程：

    • 监控的对象是主机，多个主机可以归并到一个主机组中，每个主机中有多个监控项（item）
    • 工作在主机上的zabbix agent代理，收集item的数据，将数据主动发送或被动发送给zabbix Server。
    • Server端在接收到数据后，将采集的数据与指定的阈值进行比较，如果超出阈值的范围：
      • 触动触发器，触发器产生事件events
      • 出现事件，采取行动action： - 先执行事先定义的远程命令，进行故障处理 - 如果没有恢复，则发送报警信息，发短信或邮件
    • 没有超出阈值的范围：
      • 将每次采集来的数据存储到数据库中，每次收到采集的数据都进行存储。
    • 一定时间后，将这段时间采集来的数据进行数据整合，也就是生成这段时间内的聚合数据。
    • zabbix web-GUI将通过PHP程序，从数据库中取出采集到的数据，进行展示，可以实时展示存储到数据库的实时数据，也可以展示聚合数据。
