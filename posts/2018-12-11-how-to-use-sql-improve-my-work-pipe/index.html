在这篇博文中，我想分享两个工具： sql 和 chart ，我们最近一直在使用它们.它们共同组成了一个有趣的模式，用于在终端内单独或同时修改多个数据库，可以非常方便的可视化数据.

MySQL?

然而，在使用MySQL时，多年来我们缺乏这些实际的ad-hoc交互.为什么MySQL不一样？

MySQL客户端主要不是为了可组合性，：

  1. 我们感兴趣的数据库通常在远程机器上运行
  2. MySQL客户端是可交互式的
  3. 它以一种易于阅读但难以管道化的格式输出结果

这是在MariaDB客户端上运行的查询：

> SELECT * FROM (SELECT 1, 2, 3) t;
+---+---+---+
| 1 | 2 | 3 |
+---+---+---+
| 1 | 2 | 3 |
+---+---+---+
1 row in set (0.00 sec)

我们可以解决这些障碍：

  1. mysql显然允许我们连接到远程数据库，前提是你在MySQL服务器内向用户授予适当的权限，并允许服务器防火墙上的远程传入连接.

    $ mysql -h remote.movio.co -P3306 -u mariano -pmypass db

  2. 我们可以使用 -e 标志向客户端提供查询.提供此标志时，客户端不是交互式的.

    $ mysql -h remote.movio.co -P3306 -u mariano -pmypass db \
-e"SELECT * FROM (SELECT 1, 2, 3) t"
+---+---+---+ 
| 1 | 2 | 3 | 
+---+---+---+ 
| 1 | 2 | 3 | 
+---+---+---+

  3. 要获得我们可以使用的输出，我们可以添加 -N（跳过列名称），-s（静默，少输出）和 -r（原始输出，表格格式）标志.

    $ mysql -h remote.movio.co -P3306 -u mariano -pmypass db \
-Nsre"SELECT * FROM (SELECT 1, 2, 3) t"
1 2 3

  4. 最后，作为一个例子，我们现在可以将输出传递给 tr 和 bc 来将它们结合起来：

    $ mysql -h remote.movio.co -P3306 -u mariano -pmypass db \
-Nsre"SELECT * FROM (SELECT 1, 2, 3) t" | tr "\t" "+" | bc
6

即使我们现在可以在任何查询的输出上应用整个 coreutils 工具箱，但这种方法仍然存在两个重要问题：

  1. 上一行太长，不便于即时查询.
  2. 生产数据库的密码频繁出现在 bash 历史文件.

我们可以用bash函数解决这两个问题：

$ function m() {
   mysql -h remote.movio.co -P3306 -u mariano -pmypass db -Nsre"$1"
}

这样就可以使用下面的命令：

$ m 'SELECT * FROM (SELECT 1, 2, 3) t' | tr "\t" "+" | bc
6

但这个想法只适用于特定的数据库;如果我们想查询另一个数据库，我们需要写另一个函数或增加至少一个参数.

初探 sql

为了解决上面众多问题，我们构建了一个简单的基于 Go 语言的 Cli 工具 – sql.

sql 从标准输入读取，直到 EOF （它假定内容是MySQL查询），然后将查询发送到一个或多个数据库，并以表格格式输出结果.

使用 sql 工具之后，我们前面的例子就变成了：

$ echo "SELECT * FROM (SELECT 1, 2, 3) t" | sql remote | tr "\t" "+" | bc
6

sql 需要一个填有 数据库位置 和 凭据 的配置文件.前面的示例假设你有一个如下所示的配置文件：

{
 "remote": {
   "dbName": "db",
   "dbServer": "remote.movio.co",
   "pass": "mypass",
   "user": "mariano"
 }
}

译者：该配置文件可以填写多个数据库的位置以及凭据

它可以并行查询多个数据库（使用goroutines）.要在查询多个数据库（不一定排序）时消除输出歧义，将数据库名称添加为每行的第一列.

$ sql remote1 remote2 <<< "SELECT name FROM red_squad ORDER BY id DESC LIMIT 3"
remote1    Lucian
remote2    Isaac
remote1    Gary
remote1    Mariano
remote2    Mikhail
remote2    Am

使用 sql all 可以查询配置文件中的所有数据库.

此外，它支持本地计算机无权远程连接但你具有 ssh 访问服务器的情况;在这种情况下， sql 将首先 ssh 到服务器，然后连接到 mysql .

注意: sql 依赖于mysql客户端和可选的 ssh .使用 ssh 时，必须在远程服务器上安装 mysql 客户端.

接下来遇到的问题

虽然这个闪亮的新工具在某些情况下使查询更快更方便（例如，不需要 ssh 到远程主机，不必键入主机名和用户名/密码），它还启用了一些重要的新工具 - 比如能够比较或在我们的许多或所有数据库中断言.

在这样做的过程中，我们发现，有时，查询的结果，无论是否是多重的，都不容易在纯文本中解释.有时图表比人类的数据表更好.

下面是我们遇到了一个例子

在Movio，我们每个月都会发送数亿封电子邮件和短信.每隔一段时间，一些营销活动出现问题，我们需要快速弄清楚发生了什么.

比如说，我们收到一封电子邮件，通过支持关于营销活动的不寻常的可传递性结果，电影参展商名为“ABC”.

如果问题是未能执行营销活动，或者在发送通信时遇到问题，我们会立即在我们的广告系列基础架构上查找提醒，并密切关注我们的监控信息中心上的可疑活动.但是，在这种情况下，广告系列似乎正在执行，但结果并不像预期的那样.

我将为此次调查提出三个步骤：

  1. 这里要做的第一件事是问题的范围;这种情况发生在每个客户或ABC吗？
  2. 在降低风险之后，我们需要一些背景.ABC的广告系列如何表现？这种情况真的是异常吗？
  3. 最后，我们可以真正深入研究问题的根本原因.

假设我们有下表（其中一个针对我们的每个客户，分布在几十个数据库服务器上）：

CREATE TABLE Campaigns (
   sentCount INT,
   utcSent TIMESTAMP
)

查找过去七天内发送的广告系列的结果的直观查询将是：

> SELECT sentCount, utcSent FROM Campaigns
 WHERE utcSent > DATE_SUB(UTC_TIMESTAMP(), INTERVAL 7 DAY)

当然，在每个数据库上执行此操作都需要很长时间，因此我们可以使用sql代替：

$ echo "SELECT sentCount, utcSent FROM Campaigns
 WHERE utcSent > DATE_SUB(UTC_TIMESTAMP(), INTERVAL 7 DAY)" | sql all

不幸的是，生成的数据集相对难以阅读和解释：

ABC  184063  2017-05-20 15:08:37
JKL  14161 2017-05-20 12:02:49
DEF  1510620 2017-05-20 11:02:59
DEF  16923 2017-05-17 17:58:23
ABC  63502 2017-05-17 17:57:09
MNO  245708  2017-05-17 17:48:57
GHI  16959 2017-05-17 17:32:21
...

我们需要的是散点图.有没有一种简单的方法从这个输出中获得一个？

初探 chart

Linux有一个名为 gnuplot 的世界级图表工具.它功能强大，涵盖了所有图表类型和功能，但它对于快速临时图表也略显不方便.这有点难学.

相反，我们构建了一个非常简单的图表工具，称为 chart.它从标准输入(STDIN)读取数据（例如 sql 的输出），识别格式并打开带有数据图表的浏览器选项卡.对于实际的图表，它使用 Chart.js，这是我们可以找到的最简单，最轻量级的图表库.

chart 只有一组非常有限的智能和修饰符.关键是保持简单，只实现我们使用的.

对于第一步（降低风险），我们需要每个客户上周发送计数的散点图：

  • X轴应该是日期
  • Y轴应该是发送计数
  • 应通过颜色识别不同的客户
  • 有些客户发送的通信比其他客户多，所以我们需要在Y轴上使用对数刻度来查看有用的输出

所有这些在 chart 中都非常简单：

$ echo "SELECT sentCount, utcSent FROM Campaigns" \
 "WHERE utcSent > " \
 "DATE_SUB(UTC_TIMESTAMP(), INTERVAL 7 DAY)" | sql all | \
 chart scatter log mysql

mysql 参数是–date-format的别名 “2006-01-02 15:04:05” ，并且因为它很常见而被添加

我们没有看到全面的发送计数突然下降，因此可以安全地转到第二步：为这个特定客户获取一些上下文.

$ echo 'SELECT sentCount, utcSent FROM Campaigns WHERE ' \
'utcSent > DATE_SUB(UTC_TIMESTAMP(), INTERVAL 2 MONTH) ORDER BY ' \
'utcSent DESC' | sql abc | chart line mysql --zero-based

很明显，这个特定的活动正在接近曲线的末尾.在这种情况下，我们可以使用结构化日志文件来列出发送失败.该文件如下所示：

{"utcInserted": "May 17 17:47:17", "reason": "hard_bounce", "campaignId": "3055", "customer": "MNO"}
{"utcInserted": "May 17 17:47:34", "reason": "soft_bounce", "campaignId": "123", "customer": "ABC"}
{"utcInserted": "May 17 17:48:02", "reason": "feedback", "campaignId": "58373", "customer": "DEF"}
{"utcInserted": "May 17 17:48:02", "reason": "hard_bounce", "campaignId": "123", "customer": "ABC"}
...

使用一点 jq 魔术，我们可以过滤掉我们不感兴趣的日志，并输出每次失败的原因：

$ jq -r 'select(.customer == "ABC" and .campaignId == "123")` \
`| .reason' /var/log/failures.log
soft_bounce
hard_bounce
hard_bounce
network_timeout
soft_bounce
...

标识符频率也是一种非常常见的用例，因此当图表收到字符串列表时，它假定你需要一个饼图，其中每个不同的字符串作为一个类别，其频率作为总计的一部分.

$ jq -r 'select(.customer == "ABC" and .campaignId == "123")` \
`| .reason' /var/log/failures.log | chart

硬弹跳似乎是了失败的主要原因.网络超时往往具有更系统的效果，因此这种结果并不出乎意料. 可能的解释是：

  • 我们的一个或多个邮件服务器已被列入黑名单
  • 该活动针对管理不善的地址清单
  • 电子邮件内容触发了第三方垃圾邮件检查程序

还有一些工作要做，但是我们做了一个非常全面的初步诊断，并且在终端的几行中排除了更大的潜在威胁.

该工作流程的一些优点包括：

  1. 能够一次查询所有数据库，可以排除更严重的问题.否则，我们可能只检查了特定客户的数据库，忽略了根本原因.
  2. 我们触及了它允许的表面.想想 coreutils 中丰富的工具，以及不同的图表类型可以揭示的内容.在简单的表格上可以很好地总结饼图和条形图;但折线图和散点图可以显示在人类矩阵上不可见的复杂相关性.

陷阱

如果你打算为工作流程尝试 sql 和 chart ，则应该首先阅读本节.

Read-only is best

应谨慎使用sql;特别是对多个数据库.每个人都会犯错，而sql允许你增加你的破坏力！如果你还没有，请创建具有只读权限的数据库用户名，并将其与sql一起使用.

Even read-only can be dangerous

注意： MyISAM 表将对读取执行表级锁定.即使在 InnoDB 上，也请不要在整个数据库集中选择 SELECT * FROM users！

Timing queries

sql 不像 MySQL 客户端那样输出查询运行时间.但是，你可以像这样使用时间 coreutils 工具：

$ time sql test_db <<< "SELECT COUNT(*) FROM users"
real    0m6.624s
user    0m0.020s
sys     0m0.034s

请注意，这些数字包括 sql 进程花费的整个时间，包括输出到 stdout，但MySQL客户端不包括该部分时间.

Watching SHOW PROCESSLIST

如果要仔细检查某个数据库服务器上是否正在运行查询，则 sql 为你提供了一种检查为该模式运行的查询的简便方法：

watch "echo 'show processlist' | sql my_db"

你可以进一步 grep 等待你的特定查询：

watch "echo 'show processlist' | sql my_db | grep 'FROM my_table'"

Learning chart should be very easy

有3种简单的方法可以学习如何使用它：

  1. 看看 cheatsheet.这显示了示例数据，运行的命令以及常见用例的结果图表.
  2. 多尝试一下喽. 将数据传输到图表并尝试一些可用选项.
  3. 不信任它.添加 --debug 选项以检查图表 如何解释你的输入.你会对它的运作方式有所了解.

总结

通过 sql 和 chart ，我们找到了一种方便的模式，可以快速诊断我们的问题。如果你的工作围绕数据库，我认为它可能对你有用。数据科学家可能会在 Jupyter 笔记本上使用 matplotlib 。

sql 和 chart 是免费的开源软件。我们希望你发现它们有用！如果可以，请随时帮助我们保持简单实用，并面对新的有趣挑战。感谢你阅读此博文。💋！
