我们都知道ElasticSearch是基于Lucencee实现的一个成品式的分布式文档搜索引擎，下面我会重点介绍ElasticSearch中的基本概念、工作流程以及整体架构。

下面为了方便，我将ElasticSearch简写为ES。

基本概念

  • 索引
    • 类似于SQL中的数据库的概念，可以在索引中读取、写入以及查询数据
    • ES主要在内部使用Lucence语法将数据写入索引中，并在索引中检索数据
    • ES中的索引可能在内部是由一个或者多个Lucence索引构成，具体如何划分，可能会由ES的分片，复制机制决定。
  • 文档
    • 是ES中最基本的实体/单元，在Lucence中也是如此
    • 文档是由字段构成的，而字段是由多个字段名构成，每个字段名由很多字段值构成
    • 每个文档有各自的字段集合，没有固定的模式以及强制的结构
    • 对客户端来讲，文档是一种JSON对象
  • 类型
    • 类似于SQL中数据表的概念
    • 允许用户在一个缩阴中存储多种文档类型，并为不同文档类型提供不同的映射
  • 映射
    • 映射可以看成定义在不同文档上的操作
    • 映射主要提供一些参数，决定如何去处理文档，（比如：如何将输入文本分割为词条，如何过滤特定条件下的词条）
  • 节点
    • 一个节点表示一个ES实例
    • 节点从功能和指责上可以划分为
      • 数据节点：持有数据，提供这些数据的搜索功能
      • 主节点（管理节点）：主要负责监督和控制其他节点的工作
      • 部落节点：主要负责跨集群间协作
  • 集群
    • 多个ES节点构成一个集群
    • 像分布式集群一样，主要能够处理超过单台主机的负载能力的数据量
    • 可以无间断的提供服务
  • 分片
    • 分片在多节点中是非常重要的概念，因为集群可以轻松的处理超过单台主机的负载能力的数据量，那么必不可少的就会涉及到将数据分散到不同的物理的Lucence索引上，这些Lucence索引被称为分片
    • 散布这些分片的工作叫做分片（sharding）
    • ES会自动的做分片，除此之外，用户也可以根据业务和应用需要，自定义和优化ES配置。
  • 副本
    • 主要用于解决集群访问压力过大导致的单机无法处理请求的问题，解决该问题的办法就是为每一个分片制作副本，其中要制作的分片被称作主分片，主要的工作流程如下：
  • 当主分片所在的ES节点宕机了，ES将会自动地在主分片的所有副本中选出一个作为新的主分片。

架构

  • ES是 P2P（对等）架构，不同以往的Master/slave结构，对等架构往往在很多地方和 Leader/Fellower架构类似。
  • 易于扩充新的节点到集群
  • 数据结构灵活，没有对ES中的数据结构强加限制
  • 准实时搜索和版本同步，主要解决分布式系统中的查询延时和节点之间的数据不同步问题
  • 管理节点不比其他的节点重要！

工作流程

启动流程

在启动时，ES主要使用发现机制来发现同一集群的其他节点，并与它通讯的，主要🈶以下过程：

  • ES节点向网络中发送多播请求
  • 处于同一个网络的其他节点处理该请求，并且在集群内部有个管理节点负责控制和协调ES节点的数据传输以及分片等。

与ES通信

ES提供了REST API 的访问方式，主要包括索引，推送和查询数据。

索引数据

  • 发送数据到指定的索引

    curl -XPUT http://localhost:9200/blog/article/1 \
  -d '{"title":"xxx","tags":["xx"],["xxx"]}'

  • bulk API

建立索引只能在主分片上操作而不是副本分片，当一个索引请求发送到一个节点上，而该节点并没有对应的住分片，这时这个请求将会转发到正确的主分片所在的节点上，然后主分片的节点将会把请求群发给对应副本分片的所有节点。具体流程见下图：

查询数据

查询是ES的主要任务，查询底层主要是利用Lucence，语法也比较类似。

查询主要分为两个阶段：（类似于MR或Divide & Conque过程）

  • 分散阶段： 查询分散到包含相关文档的多个分片的查询，跨节点查询
  • 合并阶段： 收集结果，做一些合并、排序和自定义的后续处理，返回给客户端
